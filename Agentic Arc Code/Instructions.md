# Instructions.md — Goal-Oriented Agent with Memory, Perception, Decision, Action (MCP) and Artifacts

> **Purpose**: This document is the **single source of truth** for building an AI solution that follows the architecture described in *Notes.txt*: a goal-iterating agent loop with **four named roles (Memory, Perception, Decision, Action)**, a parallel **Artifacts store**, and an **LLM Gateway** that enforces routing and structured outputs.  
> **Audience**: Engineers implementing the agent runtime, prompt templates, schemas, and operational guardrails.

---

## 0) Non‑Negotiable Principles

1. **Explicit decomposition into bounded goals**: The system must turn a user request into a **list of short, imperative goals** with stable order and **done flags**.  
2. **One-goal-at-a-time decisioning**: **Decision never sees other goals** besides the one currently being worked.  
3. **Memory first, every iteration**: Each iteration begins by asking: *what do we already know?*  
4. **Artifacts are bytes; Memory is metadata**: Large tool outputs are stored as artifacts; Memory stores only handles and descriptors.
5. **Perception controls attachments**: Decision receives raw artifact bytes **only if** Perception attaches them to the current goal.
6. **Strict schemas at role boundaries**: Every boundary between roles is validated (Pydantic models → JSON Schema → structured output parsing).
7. **Action is non-LLM dispatch only**: Action never calls the LLM.
8. **Context is rebuilt each loop**: Do not assume any hidden state inside the model; state is reconstructed from memory hits, history, and prior goals.

---

## 1) Runtime Architecture (High Level)

### 1.1 Roles (called in order each iteration)

- **Memory (service)**
  - Stores typed items: facts, preferences, tool outcomes, scratchpad.
  - Provides low-cost reads via keyword overlap and structured filtering.
  - Optional LLM-scored relevance used only when keyword recall is weak.

- **Perception (orchestrator; 1 LLM call / iteration)**
  - Observes: user query, memory hits, run history, prior goals.
  - Emits: updated goal list with `done` flags and optional artifact attachment.
  - Also acts as verifier: decides when goals are satisfied based on history.

- **Decision (planner; 1 LLM call / iteration if work remains)**
  - Receives exactly **one** bounded goal.
  - Outputs either:
    - a final **plain-text answer**, OR
    - a **single** MCP tool call.

- **Action (executor; 0 LLM calls)**
  - Executes exactly one MCP tool call.
  - If output size exceeds threshold (default: **4 KB**), writes bytes to Artifacts store.
  - Writes tool outcome to Memory.

### 1.2 Supporting Components

- **Artifacts store** (content-addressable)
  - Stores bytes as `art:<sha256-prefix>`
  - Memory stores `(artifact_id, descriptor)`; raw bytes are attached to Decision only via Perception.

- **LLM Gateway (V3)**
  - Every LLM call routes through the gateway.
  - Perception calls are pinned to a reliable provider (e.g., `provider="g"`).
  - Decision and Memory (classifier) use `auto_route` unless pinned.
  - Enforces structured output via `response_format` validation.

## 2) Data Contracts (Schemas)

> **Implementation requirement**: define these as Pydantic models in `schemas.py`, and use their JSON Schema as the `response_format` for Perception and the Memory classifier.

### 2.1 MemoryItem

- `id: str`
- `kind: Literal["fact", "preference", "tool_outcome", "scratchpad"]`
- `keywords: list[str]`
- `descriptor: str`  *(one short human-readable line)*
- `value: dict`      *(structured payload)*
- `artifact_id: str | None`
- `source: str`
- `run_id: str`
- `goal_id: str | None`
- `confidence: float`
- `created_at: datetime`

### 2.2 Artifact

- `id: str` *(format: `art:<sha256-prefix>`)*
- `content_type: str`
- `size_bytes: int`
- `source: str`
- `descriptor: str`

### 2.3 Goal

- `id: str` *(generated by the loop; stable per run)*
- `text: str` *(short imperative)*
- `done: bool`
- `attach_artifact_id: str | None` *(set only by Perception)*

### 2.4 Observation (Perception output)

- `goals: list[Goal]`

### 2.5 ToolCall

- `name: str`
- `arguments: dict`

### 2.6 DecisionOutput (Decision output)

- Exactly one is populated:
  - `answer: str | None`
  - `tool_call: ToolCall | None`

## 3) Persistent State & Filesystem Layout

**Minimum required directories:**

- `state/`
  - `memory.json` *(durable; reused across runs)*
  - `artifacts/`
    - `art:<sha>.bin` *(raw bytes)*
    - `art:<sha>.json` *(Artifact metadata)*

**Persistence rules:**

- Load `state/memory.json` on first Memory read.
- Write back after every mutation:
  - `remember(...)` (LLM classification write)
  - `record_outcome(...)` (tool outcome write)
- Clearing `state/` resets the agent.

## 4) Memory Service (Required Behaviors)

### 4.1 Reads (no LLM by default)

1. `memory.read(query, history, kinds=None, top_k=8) -> list[MemoryItem]`
   - Use keyword overlap scoring over:
     - MemoryItem.keywords
     - tokens from MemoryItem.descriptor
   - Apply stopword removal, lowercase tokenization.

2. `memory.filter(kinds=None, goal_id=None, recent=None) -> list[MemoryItem]`
   - Structured filter only.

3. `memory.relevant(query, kinds=None, top_k=5) -> list[MemoryItem]`
   - Optional: LLM-scored relevance over a candidate pool.
   - Use only if keyword recall confidence is low.

### 4.2 Writes

1. `memory.remember(raw_text, source, run_id, goal_id=None) -> MemoryItem`
   - **LLM classification required**.
   - Must:
     - choose `kind`
     - extract structured `value`
     - generate `descriptor`
     - generate `keywords`
     - set `confidence`
   - Store result in memory.json.

2. `memory.record_outcome(tool_call, result_text, artifact_id, source, run_id, goal_id) -> MemoryItem`
   - No LLM.
   - Set `kind="tool_outcome"`.
   - Keywords are derived from:
     - tool name
     - argument tokens
     - short tokens from descriptor

### 4.3 Memory Classifier Prompt (Structured Output)

Use this system instruction (template) for the classifier call:

- **Input**: `raw_text`, `source`, `run_id`, optional `goal_id`
- **Output**: a `MemoryItem` JSON strictly matching schema.

Classifier rules:

- If the text asserts a durable truth (dates, attributes, relationships), choose **fact**.
- If the user expresses a preference, choose **preference**.
- If it describes a tool dispatch result, choose **tool_outcome** (normally created by code).
- If it is ephemeral working state, choose **scratchpad**.

Self-check before emitting:

- Ensure `keywords` are short, meaningful, and overlap likely future queries.
- Ensure `descriptor` is a single concise sentence.
- Ensure `value` is structured (not just a copied paragraph).

## 5) Perception Role (Observe → Goals)

### 5.1 Perception Inputs

Per iteration, call:

`observe(query, hits, history, prior_goals, run_id) -> Observation`

Where:

- `query: str` *(original user query for the run)*
- `hits: list[MemoryItem]` *(top-k memory items relevant to the query)*
- `history: list[dict]` *(accumulated events)*
- `prior_goals: list[Goal]` *(goals from previous iteration; empty on first iteration)*

### 5.2 Perception Obligations

Perception must satisfy **all**:

1. **Goal decomposition** (only when `prior_goals` is empty)
   - Produce 1..N goals.
   - Each goal must be bounded and executable.

2. **Sticky done flags**
   - A goal becomes done as soon as history shows it was satisfied.
   - Once done, it must remain done.

3. **Attachment control**
   - For the **first unfinished goal only**, decide if raw bytes from any artifact are required.
   - If yes, set `attach_artifact_id` to a handle that exists in hits.

4. **Goal order stability**
   - Do not reorder goals.
   - Do not insert new goals in the middle.
   - Do not drop goals.

### 5.3 Perception Prompt Requirements

Perception must be called with:

- `response_format` set to Observation JSON Schema.
- A provider override to a reliable model (e.g., `provider="g"`).
- Temperature recommended: **1.0** (avoids repetitive looping in some providers).

### 5.4 Artifact Indexing Safety (Recommended)

To reduce hallucinated artifact handles:

- Present hits containing artifacts to Perception as an indexed list:
  - `artifact_index: i`
  - `artifact_id: art:...`
  - `descriptor`
- Perception outputs an integer index (or selects from available ids).
- The outer loop maps index → real `artifact_id`.

### 5.5 Perception Self-Checks

Before emitting Observation:

- Confirm each goal is **imperative**, **bounded**, and **not redundant**.
- Confirm no `attach_artifact_id` is set for a done goal.
- Confirm at most one goal has an attachment (the first unfinished).
- Confirm goal count and ordering match `prior_goals` when provided.

## 6) Decision Role (Next Step for One Goal)

### 6.1 Decision Inputs

`next_step(goal, hits, attached, history, mcp_tools) -> DecisionOutput`

- `goal: Goal` *(only the current goal)*
- `hits: list[MemoryItem]` *(relevant memory items)*
- `attached: list[tuple[artifact_id, bytes]]` *(only if Perception attached)*
- `history: list[dict]` *(recent events; last N recommended)*
- `mcp_tools: list[dict]` *(tool specs presented to the model)*

### 6.2 Output Rule (Hard Constraint)

Decision must output **exactly one** of:

- `answer` (plain text)
- `tool_call` (exactly one tool invocation)

Decision must **never**:

- call multiple tools in one response
- return both answer and tool call
- narrate tool execution

### 6.3 Artifact Handle Rule (Hard Constraint)

- Strings beginning with `art:` are **internal handles**.
- Decision must **never** pass `art:` strings as tool arguments.
- If artifact content is needed, it will appear under **ATTACHED ARTIFACTS**.

### 6.4 Substantive Answer Rule

When Decision answers (no tool):

- Provide at least **3 sentences** OR a **bullet/numbered list**.
- Do the actual work (extract, compare, decide), not a meta response.

### 6.5 Decision Reasoning Type Tags (for reliability)

Decision must internally choose the reasoning type that best applies:

- `arithmetic` (numeric calculations)
- `logic` (constraints, if/then)
- `lookup` (requires external retrieval)
- `synthesis` (merge multiple sources)
- `planning` (sequence actions)
- `classification` (map input to categories)

**Implementation**: Do not ask the model to reveal chain-of-thought.  
Instead, require a short `REASONING_TYPE: <one of above>` line in a separate field **only if** you store it in history for debugging (optional). If you do not store it, keep it as an internal instruction.

### 6.6 Decision Self-Checks

Before emitting DecisionOutput:

- If goal cannot be answered with current info, choose a tool.
- If choosing a tool:
  - confirm tool exists in `mcp_tools`
  - confirm required arguments are provided
  - confirm no `art:` arguments
- If answering:
  - confirm answer addresses goal directly
  - confirm answer is substantive (3+ sentences or list)

### 6.7 Decision Fallbacks

If uncertain:

- Prefer a tool call that reduces uncertainty (e.g., `web_search`, `fetch_url`, `read_file`).
- If no tool can help, answer with:
  - what is known
  - what is unknown
  - what additional input is needed

## 7) Action Role (MCP Dispatch + Artifact Handling)

### 7.1 Dispatch Rules

Action receives a `ToolCall` and:

1. Validates tool name exists.
2. Validates no argument value begins with `art:` (guardrail).
3. Calls MCP: `session.call_tool(name, arguments=...)`.
4. Collapses content blocks to text.
5. If payload size > **4 KB**:
   - write bytes to artifacts store
   - return `(descriptor_preview, artifact_id)`
6. Else:
   - return `(result_text, None)`

### 7.2 Error Handling

If MCP fails:

- Return a short error descriptor (do not throw unhandled exceptions).
- Record the failure in history.
- Allow Perception to decide whether:
  - retry
  - choose alternate tool
  - mark goal blocked and proceed

## 8) Main Agent Loop (Required Control Flow)

### 8.1 Loop Skeleton

1. Generate `run_id`.
2. Initialize `history=[]`, `prior_goals=[]`.
3. **Durable memory contract**: `memory.remember(user_query, source="user_query", run_id=run_id)`.
4. Create MCP session and load tool specs.
5. For `it in 1..MAX_ITERATIONS`:
   1. `hits = memory.read(query, history)`
   2. `obs = perception.observe(query, hits, history, prior_goals, run_id)`
   3. Update `prior_goals = obs.goals`
   4. If all goals done: break.
   5. Let `goal = first unfinished goal`.
   6. If `goal.attach_artifact_id` exists and artifact exists:
      - load bytes, pass to Decision under **ATTACHED ARTIFACTS**.
   7. `out = decision.next_step(goal, hits, attached, history, tools)`
   8. If `out.answer`:
      - append answer event to history
      - continue
   9. Else:
      - `result_text, art_id = action.execute(out.tool_call)`
      - `memory.record_outcome(...)`
      - append action event to history
6. Return `final_answer_from(history)`

### 8.2 History Event Shapes (Recommended)

Store events as dicts mirroring typed boundaries:

- Answer event:
  - `{"iter": it, "kind": "answer", "goal_id": goal.id, "text": out.answer}`

- Action event:
  - `{"iter": it, "kind": "action", "goal_id": goal.id, "tool": name, "arguments": args, "result_descriptor": preview, "artifact_id": art_id}`

## 9) MCP Tooling Requirements

### 9.1 Minimal Tool Set

The architecture assumes a tool server exposing (names may vary):

- `web_search({query})`
- `fetch_url({url})`
- `get_time({tz?})`
- `currency_convert({amount, from, to})`
- `read_file({path})`
- `list_dir({path})`
- `create_file({path, content})`
- `update_file({path, content})`
- `edit_file({path, edits})`

### 9.2 Tool Selection Guidance

Decision should choose tools when:

- external info is required (`web_search`, `fetch_url`)
- local data is needed (`read_file`, `list_dir`)
- an artifact must be created/updated (`create_file`, `update_file`, `edit_file`)

## 11) Prompt Packs (Copy/Paste Templates)

> These are **templates**. Implementers should inject run-time values for query, hits, history, tools, and attached artifacts.

### 11.1 Perception — System Prompt (template)

You are **Perception**, the orchestrator.

Your job each iteration:

1) If `prior_goals` is empty, decompose the user query into 1..N bounded goals (short imperative statements).  
2) For each existing goal, inspect the history and set `done=true` exactly when the history contains evidence the goal is satisfied. Once done, it stays done.  
3) For the **first unfinished goal only**, decide whether you must attach raw bytes from an artifact found in `hits`. If yes, set `attach_artifact_id` to an artifact that exists.  
4) Preserve goal order and count from `prior_goals`. Do not reorder, insert, or delete goals.

Self-checks before output:

- goal order stable
- done flags sticky
- attachment only on first unfinished goal
- attachment references an artifact present in hits

Output must match the Observation JSON schema.

### 11.2 Decision — System Prompt (template)

You are **Decision**.

You receive exactly one goal. Choose the next step:

- If you can fully satisfy the goal with the information provided, return a **plain-text answer**.
- Otherwise, return exactly **one** tool call from the provided MCP tools.

Hard rules:

- Output must be exactly one of: `{answer}` or `{tool_call}`.
- Never call more than one tool.
- Never pass `art:` handles as tool arguments.
- If artifact bytes are needed, read them only from the **ATTACHED ARTIFACTS** section.

Quality rules:

- Answers must be substantive: 3+ sentences or a list.
- Prefer tools when uncertain.
- If no tool can reduce uncertainty, state what is known, what is unknown, and what user input is required.

### 11.3 Memory Classifier — System Prompt (template)

You are **MemoryClassifier**.

Task: Convert `raw_text` into a single `MemoryItem`.

Rules:

- Choose `kind` from: fact | preference | scratchpad | tool_outcome.
- Extract a structured `value` dict.
- Provide a short `descriptor` line.
- Provide 4–12 `keywords` likely to match future queries.
- Provide a `confidence` between 0 and 1.

Self-check:

- descriptor is 1 line
- value is structured, not a paragraph
- keywords are meaningful and not stopwords

Output must match the MemoryItem JSON schema.

## 12) Built-in Self-Test Scenarios (Acceptance Tests)

> Use these as end-to-end tests for the loop. They cover: artifact attach, multi-goal planning, durable memory across runs, and multi-source synthesis.

### Test A — Artifact Attach

**Query**: Fetch a web page (e.g., Wikipedia) and extract specific facts.

**Expected**:

- Iteration 1: Decision calls `fetch_url`.
- Action stores large payload as artifact.
- Iteration 2: Perception attaches artifact to extraction goal.
- Decision answers using attached bytes.

### Test B — Multi-goal + Constraint

**Query**: Find activities + fetch weather + decide best option.

**Expected**:

- Tools used: `web_search`, then `fetch_url` (or similar).
- Final decision uses weather fact carried via Memory/History.

### Test C — Durable Memory Across Runs

**Run 1**: “My mom’s birthday is 15 May 2026. Remember that…”

**Run 2**: “When is mom’s birthday?”

**Expected**:

- Run 1 creates a fact MemoryItem.
- Run 2 answers from memory hits without re-asking.

### Test D — Multi-source Synthesis

**Query**: Search a topic, fetch top N results, summarise common advice.

**Expected**:

- Artifacts created for fetched pages.
- Perception attaches at least one relevant artifact to synthesis goal.
- Decision produces numbered list of agreed points.

## 13) Prompt-Quality Checklist (Aligned to Prompt Validator)

This Instructions.md is designed to satisfy the following prompt-quality requirements:

- **Explicit reasoning instructions**: roles have step-by-step obligations and self-checks.
- **Structured output**: Perception and Memory use strict JSON schema; Decision uses a strict either/or output.
- **Tool separation**: Decision plans; Action executes; artifact handles are blocked from tools.
- **Conversation loop support**: iteration protocol with history + prior goals + memory hits.
- **Instructional framing**: copy/paste prompt templates + acceptance tests.
- **Internal self-checks**: each role has pre-output checklists.
- **Reasoning type awareness**: Decision includes reasoning type categories (without revealing chain-of-thought).
- **Fallbacks**: uncertainty rules and error handling for tool failures.

## 14) Implementation Notes (Do / Don’t)

### Do

- Do validate every model output against its schema.
- Do keep Memory reads cheap; pay LLM cost at write time.
- Do keep artifact bytes out of normal prompts.
- Do let Perception decide when goals are satisfied.

### Don’t

- Don’t let Decision see the entire goal list.
- Don’t store large tool payloads inside Memory.
- Don’t allow `art:` handles to leak into tool arguments.
- Don’t rely on a separate verifier if Perception already re-evaluates goals each iteration.
