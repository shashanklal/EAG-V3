📚 LLM Gateway · Help

Dashboard Help

This service routes LLM calls across seven Databricks Model Serving endpoints, with automatic failover, rate-limit tracking, persistent logging, and a manual override. Below: how to use it and what each endpoint offers.

Using the gateway

Python (built-in client)

from client import LLM, ask

# one-shot
print(ask("Explain transformers in 3 lines"))

# explicit provider (shortcut keys: l, g, m, q, c, gpt, c6)
print(ask("hi", provider="q"))    # qwen3_80b
print(ask("hi", provider="c"))    # claude_s4

# full client
llm = LLM()
r = llm.chat("hi", provider="claude_s4", max_tokens=128, temperature=0.5)
print(r["text"], r["provider"], r["latency_ms"])

# streaming
for chunk in llm.stream("count to 5"):
    print(chunk, end="", flush=True)

HTTP (OpenAI-style)

curl http://localhost:8101/v1/chat \
  -H "Content-Type: application/json" \
  -d '{"prompt": "hello", "provider": "c", "max_tokens": 128}'

Provider overview

All models are served via Azure Databricks Model Serving at a single workspace endpoint. Authentication uses a Databricks Personal Access Token (PAT).

1. llama31_8b — Meta Llama 3.1 8B Instruct

  • Endpoint: databricks-meta-llama-3-1-8b-instruct
  • Context: 8K tokens
  • Capabilities: tools, structured output
  • Best for: fast routing, simple queries, low-latency responses

2. gemma3_12b — Google Gemma 3 12B

  • Endpoint: databricks-gemma-3-12b
  • Context: 8K tokens
  • Capabilities: tools, structured output
  • Best for: lightweight general tasks, routing decisions

3. llama4_mav — Meta Llama 4 Maverick

  • Endpoint: databricks-llama-4-maverick
  • Context: 128K tokens
  • Capabilities: tools, structured output, parallel tools
  • Best for: long-context tasks, complex multi-step workflows

4. qwen3_80b — Qwen3 Next 80B

  • Endpoint: databricks-qwen3-next-80b-a3b-instruct
  • Context: 32K tokens
  • Capabilities: tools, structured output, reasoning, parallel tools
  • Best for: reasoning tasks, code generation, analytical work

5. claude_s4 — Claude Sonnet 4

  • Endpoint: databricks-claude-sonnet-4
  • Context: 200K tokens
  • Capabilities: tools, structured output, reasoning, parallel tools
  • Best for: large-context tasks, complex reasoning, high-quality output

6. gpt_oss — GPT-OSS 120B

  • Endpoint: databricks-gpt-oss-120b
  • Context: 128K tokens
  • Capabilities: tools, structured output, reasoning, parallel tools
  • Best for: general-purpose high-quality generation, complex tasks

7. claude_s46 — Claude Sonnet 4.6

  • Endpoint: databricks-claude-sonnet-4-6
  • Context: 200K tokens
  • Capabilities: tools, structured output, reasoning, parallel tools
  • Best for: highest-quality output, large-context reasoning, most capable model

Configuration (.env)

The gateway reads from .env (parent directory). Only DATABRICKS_TOKEN is required — all endpoints share the same token and workspace.

# Databricks workspace
DATABRICKS_HOST=https://adb-2177732704131972.12.azuredatabricks.net
DATABRICKS_TOKEN=dapi...

# Optional: override default model for any provider
DATABRICKS_MODEL_LLAMA31_8B=databricks-meta-llama-3-1-8b-instruct
DATABRICKS_MODEL_GEMMA3_12B=databricks-gemma-3-12b
DATABRICKS_MODEL_LLAMA4_MAV=databricks-llama-4-maverick
DATABRICKS_MODEL_QWEN3_80B=databricks-qwen3-next-80b-a3b-instruct
DATABRICKS_MODEL_CLAUDE_S4=databricks-claude-sonnet-4
DATABRICKS_MODEL_GPT_OSS=databricks-gpt-oss-120b
DATABRICKS_MODEL_CLAUDE_S46=databricks-claude-sonnet-4-6

# Failover order and port
LLM_ORDER=llama31_8b,gemma3_12b,llama4_mav,qwen3_80b,claude_s4,gpt_oss,claude_s46
GATEWAY_V3_PORT=8101

How failover works

  1. Caller sends request (with or without provider).
  2. Router estimates token usage from prompt length.
  3. Walks providers in LLM_ORDER, skipping any that:
    • Are inside their per-provider cooldown window (cooldown = 60/RPM)
    • Have hit RPM, RPD, TPM, or daily token cap
    • Have a max_ctx smaller than the prompt
  4. First eligible provider gets the call. On 5xx / timeout / 429 → moves to next.
  5. If provider=... was explicitly set, no failover — error surfaces.
  6. Every call (success or fail) is logged to gateway.db with provider, model, tokens, latency, status.
